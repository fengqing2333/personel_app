# OpenSpec 与 Superpowers 协作规范

## 总则

**不得因任何侥幸心理跳过强制步骤。** 以下场景均不构成跳过理由：
- "这个 Bug 很简单，不需要写测试"
- "只改了一行注释，不需要审查"
- "这个是紧急修复，先合并再说"
- "上次也是这样改的，没问题"
- "测试用例太多了，跑不过来"

**任何跳过强制步骤的行为，必须由用户在对话中明确授权，且事后补全所有遗漏步骤。** Claude 不得自行决定跳过。

---

## 三条铁律

```
铁律一：Bug 修复必须失败用例先行
  → 禁止不写测试直接修复代码

铁律二：归档前必须代码审查
  → 禁止跳过审查直接归档

铁律三：审查发现问题必须创建新提案修复
  → 禁止在原变更中修补主要/阻断问题
```

**以上三条铁律在本项目强制执行，无例外。**

---

## 核心理念

```
OpenSpec     → 规定"做什么"（需求规格层）
Superpowers  → 确保"做好"（工程流程层）
```

- **OpenSpec** 产出持久化变更制品：`proposal.md` → `design.md` → `specs/` → `tasks.md`
- **Superpowers** 产出工程流程制品：设计方案、实施计划、验证证据、项目记忆

---

## 协作流程

### 完整流程

```
┌──────────────────────────────────────────────────────────┐
│ Phase 1: 探索 (Superpowers)                               │
│   ├── 读取项目记忆和环境                                    │
│   ├── 逐条澄清需求                                          │
│   └── 输出：2-3 方案对比 + 推荐方向 + 设计草案               │
│   交付至：docs/superpowers/specs/                           │
└──────────────────────┬───────────────────────────────────┘
                       ↓ 方向确认
┌──────────────────────────────────────────────────────────┐
│ Phase 2: 规格 (OpenSpec)                                  │
│   ├── /opsx:new 创建变更脚手架                             │
│   ├── proposal.md  → 为什么做、做什么                       │
│   ├── design.md    → 技术方案、权衡                        │
│   ├── specs/       → 规范性需求、场景                      │
│   └── tasks.md     → 可执行实施清单                        │
│   交付至：openspec/changes/<change-name>/                   │
└──────────────────────┬───────────────────────────────────┘
                       ↓ 用户审批
┌──────────────────────────────────────────────────────────┐
│ Phase 3: 执行 (Superpowers)                               │
│   ├── 编写实施计划到 docs/superpowers/plans/               │
│   ├── 创建隔离工作区 (worktree)                             │
│   ├── TDD: RED → GREEN → REFACTOR                         │
│   ├── Bug 修复：必须先写失败用例，再写修复代码               │
│   └── 回归验证（必须输出新鲜命令结果）                       │
└──────────────────────┬───────────────────────────────────┘
                       ↓ 验证通过
┌──────────────────────────────────────────────────────────┐
│ Phase 3.5: 代码审查 (Code Review)                         │
│   ├── 审查是归档的前置条件，不可跳过                         │
│   ├── 发现主要/阻断问题 → 创建新提案，返回 Phase 2          │
│   ├── 发现次要问题 → 就地修复，仍需通过审查                  │
│   └── 审查通过 → 进入归档                                 │
└──────────────────────┬───────────────────────────────────┘
                       ↓ 审查通过
┌──────────────────────────────────────────────────────────┐
│ Phase 4: 归档 (OpenSpec)                                  │
│   ├── /opsx:archive 归档变更、合并主规范                    │
│   └── run-superpowers-memory-closeout (如启用)             │
└──────────────────────────────────────────────────────────┘
```

### 路径规则

**小变更（需求明确）：**
```
/opsx:propose → /opsx:apply → 代码审查 → /opsx:archive
```

**复杂变更（需求模糊 / 跨模块影响）：**
```
Superpowers 探索 → OpenSpec 规格 → Superpowers 执行
  → 代码审查 → OpenSpec 归档
```

两条路径都必须经过代码审查。无例外。

## 阶段闸门 (Phase Gates)

每个 Phase 必须满足以下"进入前必须存在 / 退出前必须存在"的硬制品要求。**AI 检测到下一阶段闸门所需制品缺失时，必须停下并提示用户，不得继续执行。**

### 闸门表

| Phase | 进入前必须存在 | 退出前必须存在 |
|------|---------------|---------------|
| Phase 1 探索 | （无） | `docs/superpowers/specs/<change>.md` 草案（复杂变更必需；小变更可省略） |
| Phase 2 规格 | （Phase 1 制品，如适用） | `proposal.md` + `design.md` + `tasks.md` 全部存在且非空，`specs/` 目录存在且非空 |
| Phase 3 执行 | `tasks.md` 存在 | `tasks.md` 全部勾选 + 测试输出粘贴在 `review.md` 或 commit message |
| Phase 3.5 审查 | 测试全绿证据 | `review.md` 存在，且"结论"段已勾选"通过" |
| Phase 4 归档 | `review.md` 通过 | 主规范 `openspec/specs/` 已合并 |

### 违反闸门时的处置

1. **AI 必须停止当前操作，显式告知用户缺失的制品**，列出当前 Phase 的退出要求与缺漏文件路径
2. **用户强行要求继续时，必须在对话中明确授权**，且事后补全所有遗漏制品
3. **跳过闸门的行为在 `review.md` 中必须留痕**，作为本次变更的偏离记录

### 闸门与自动化校验的关系

闸门表是"机器可校验"的核心依据。仓库提供 `scripts/check-workflow.mjs` 脚本（详见"自动化校验"章节）按本表逐项核对，归档前必须通过。

## 铁律一：Bug 修复必须失败用例先行

### 强制步骤

```
第 1 步：定位 Bug，确认根因
第 2 步：编写失败测试用例（RED）        → 必须写，不可跳过
第 3 步：执行测试，确认测试确实失败      → 验证测试本身有效
第 4 步：编写修复代码使用例通过（GREEN）
第 5 步：回归验证                           → 按变更范围选择层级（见下方规则）
第 6 步：重构优化（REFACTOR）
第 7 步：进入代码审查
```

### 回归验证分层规则

所有变更在修复后必须执行回归测试，根据变更范围选择合适层级：

| 层级 | 名称 | 适用范围 | 命令 | 要求 |
|------|------|----------|------|------|
| **L1** | 黄金用例 | PATCH 修复（小 bug、样式调整、文案修改） | `npm run test:golden` | 秒级执行（< 5 秒），覆盖核心路径 |
| **L2** | 全量用例 | MAJOR/MINOR 变更（新功能、重构、大修复） | `npm test` 或 `npx vitest run` | 完整回归，必须全部通过 |

**判定规则：**
- 不确定时执行全量回归（L2）
- L1 通过后如果波及范围超出预期，追加 L2
- L1 失败不得提交，必须先修复再回归

### 黄金用例（L1）规范

黄金用例是核心功能的精简测试集，要求：
- **秒级执行**（< 5 秒）
- **覆盖核心路径**：每个关键功能至少一个典型用例 + 一个边界用例
- **文件命名约定**：使用 `*.golden.test.js` 后缀，与常规 `*.test.js` 并存

**执行命令**：
```bash
npm run test:golden    # 仅运行黄金用例
npm test               # 运行全量测试（含黄金用例）
```

对应 `package.json` 脚本：
```json
"test:golden": "vitest run \"**/*.golden.test.js\""
```

**兼容性**：现有 `*.test.js` 不需要立即改名；黄金用例可从现有测试中按需派生为独立 `*.golden.test.js` 文件，原文件保留作为完整测试集。

### 禁止行为

- 禁止不写测试直接修复
- 禁止先修复代码再补测试
- 禁止只做人工验证，不写自动化测试
- 禁止跳过 RED 步骤直接写修复代码
- 禁止 L1 未通过时提交代码
- 禁止小修复跳过 L1、大变更跳过 L2

### 异常处理

如果现有测试框架不支持自动化测试（如尚无测试框架），必须先建立测试基础设施，再按上述步骤修复。

## 铁律二：归档前必须代码审查

### 触发条件

**所有变更**在归档前必须经过代码审查。即使只改了一行代码。

### 审查流程

```
第 1 步：执行 /superpowers:requesting-code-review 发起审查
第 2 步：按以下标准分类问题：
   - 阻断（Blocker）— 功能错误、安全漏洞、数据丢失
   - 主要（Major）— 设计缺陷、边界遗漏、测试覆盖不足
   - 次要（Minor）— 命名、风格、注释
第 3 步：根据分类执行对应处理
```

### 处理规则

| 审查结论 | 必须执行 |
|----------|----------|
| 无问题 | 进入 Phase 4 归档 |
| 次要问题 | 就地修复，修复后仍需通过审查，然后归档 |
| 主要/阻断问题 | 不得在原变更中修补，必须创建新提案修复（见铁律三） |

### review.md 模板与强制要求

**每个变更目录归档前必须包含 `review.md`，采用统一模板。** `/opsx:archive`（或手动归档）在归档前必须检测当前变更目录下 `review.md` 是否存在且"结论"段已勾选"通过"；任一不满足，归档拒绝执行。

模板见 [`openspec/templates/review.md`](../../templates/review.md)，核心结构：

```markdown
# Code Review — v<version>-<name>

- 审查时间:
- 审查范围: (commit 哈希列表 或 受影响文件清单)

## 检查清单

### 功能与测试
- [ ] 测试覆盖关键路径与边界场景
- [ ] 黄金用例 (L1) 通过 — 命令输出
- [ ] 全量测试 (L2) 通过 — 仅 MAJOR/MINOR 必填

### 规约同步（核心）
- [ ] 本次涉及的功能在 `specs/` 中均有对应条目
- [ ] `src/` 实现行为与 `specs/` 一致（关键场景逐条对照）
- [ ] tasks.md 所有项已勾选，且每项均能在 commit 中找到对应改动

### 代码质量
- [ ] 无遗留 TODO / console.log / 调试代码
- [ ] 命名 / 风格符合既有约定
- [ ] 无安全 / 性能明显隐患

## 发现问题
- Blocker:
- Major:
- Minor:

## 结论
- [ ] 通过 → 进入归档
- [ ] 就地修复（仅 Minor）→ 修复后重新审查
- [ ] 创建新提案（Major / Blocker）→ 触发铁律三
```

**强制要求**:

- 归档前 `review.md` 必须存在
- "结论"段必须勾选恰好一项
- 若结论为"通过"，方可归档
- 若结论为"就地修复"，修复后必须重新审查（更新审查时间）
- 若结论为"创建新提案"，触发铁律三流程
- "规约同步" 三项是核心必检,禁止以"全选通过"形式敷衍

## 铁律三：审查发现问题必须创建新提案修复

### 适用条件

当审查发现 **主要（Major）** 或 **阻断（Blocker）** 问题时。

### 强制流程

```
第 1 步：/opsx:new 创建新变更（命名格式：fix-review-<原变更名>）
第 2 步：完成 proposal.md → 明确要修复的问题
第 3 步：完成 design.md → 描述修复方案
第 4 步：完成 specs/ → 更新相关规范
第 5 步：完成 tasks.md → 可执行修复任务
第 6 步：用户审批
第 7 步：修复实施（必须遵循铁律一）
第 8 步：修复完成后再次进入代码审查（铁律二）
第 9 步：审查通过后归档修复变更
```

### 禁止行为

- 禁止在原变更中附加修补提交
- 禁止以"顺便修一下"的方式处理审查发现的问题
- 禁止跳过提案直接写修复代码

## 路径选择规则

| 场景 | 强制路径 | 原因 |
|------|----------|------|
| 新功能，需求明确 | OpenSpec 核心路径 | 需求已知，无需探索 |
| 新功能，需求模糊 | 完整四阶段 | 必须先探索确认需求 |
| Bug 修复，根因已知 | OpenSpec 核心路径（必须失败用例先行） | 范围可控，但测试不可少 |
| Bug 修复，根因未知 | Superpowers 探索 → OpenSpec 规格 | 必须先定位才能定方案 |
| 重构，范围明确 | OpenSpec 核心路径 | 行为不变，只需规格化 |
| 重构，影响面广 | 完整四阶段 | 必须多方案评估 |

## 职责边界

| 维度 | OpenSpec | Superpowers |
|------|----------|-------------|
| 需求定义 | ✅ proposal.md — 目标和边界 | ❌ 不参与 |
| 技术设计 | ✅ design.md — 架构决策 | ✅ 设计草案 — 多方案对比 |
| 规范标准 | ✅ specs/ — 活文档 | ❌ 不参与 |
| 任务分解 | ✅ tasks.md — 执行清单 | ✅ 实施计划 — 拆分为子任务 |
| 代码实现 | ❌ 不执行代码 | ✅ TDD 实施、失败用例先行 |
| 代码审查 | ❌ 不参与 | ✅ 审查请求、问题分级 |
| 质量验证 | ❌ 不验证 | ✅ 回归验证、输出新鲜证据 |
| 知识持久化 | ✅ 变更归档到主规范 | ✅ 项目记忆持续更新 |

## 补充规则

1. **OpenSpec 不执行代码**：`tasks.md` 完成后，实施由 Superpowers 执行
2. **Superpowers 不写需求文档**：需求必须锁定在 OpenSpec 制品中
3. **单向依赖**：Superpowers 执行以 `tasks.md` 为唯一输入，不得反向修改
4. **规格即真相**：所有偏离需求文档的实现，必须先更新 OpenSpec 制品再实施
5. **新鲜证据**：禁止无命令输出的"我觉得没问题"式完成声明。验证必须输出新鲜命令结果
6. **不可跳步**：禁止跳过审核直接写代码，禁止跳过审查直接归档，禁止跳过提案直接修补
7. **Git 提交信息必须携带版本号**：每次提交的 commit message 中必须包含当前变更的版本号前缀 `[V<MAJOR>.<MINOR>.<PATCH>]`，格式如下：
   ```
   [V1.0.0] feat: 实时工资计算功能
   [V1.0.1] fix: 修复管理页面空白渲染
   [V1.2.0] feat: 日历文字标注优化
   ```
   版本号与 `proposal.md` 标题中的版本号一致，便于从 git 历史追溯回变更提案。
7. **规约同步硬约束**：任何 `src/` 下的**行为变更**（新增/修改/删除可观测行为）必须在同一变更目录的 `specs/` 中有对应描述。如无，视为规约脱节，审查必须打回。
   - **例外**：纯重构（行为完全不变）、纯样式调整、纯注释修改 — 但必须在 `review.md` 中显式声明"无行为变更"
   - **判定原则**：
     - "行为"指用户、调用方或测试可观测的输入/输出/副作用
     - 内部实现重命名 / 私有函数拆分 / 性能优化（行为等价）不属于行为变更
     - 当无法判断是否为行为变更时，默认视为"是"，必须更新 `specs/`

## 自动化校验（新增于 v1.3.0）

**目的**：将本规范中可机器判定的硬约束（目录命名、必备制品、标题前缀、specs 非空、tasks 全勾后 review 结论一致性）从"靠 AI 自律"提升为"靠脚本兜底"。

### 脚本：`scripts/check-workflow.mjs`

- **零依赖**：仅使用 Node 内置 `fs` / `path` 模块，不引入任何 npm 包
- **触发命令**：`npm run check:workflow`
- **扫描范围**：`openspec/changes/` 下所有**未归档**的变更目录（不递归扫描 `archive/`）

### 检查项

| # | 检查项 | 严重级别 |
|---|--------|----------|
| 1 | 目录命名匹配 `v<MAJOR>-<MINOR>-<PATCH>-<kebab-name>` | ERROR |
| 2 | 必备文件 `proposal.md` / `design.md` / `tasks.md` / `.openspec.yaml` 全部存在且非空 | ERROR |
| 3 | `proposal.md` 标题以 `[V<x.y.z>]` 前缀开头，版本号与目录一致 | ERROR |
| 4 | `specs/` 目录存在且至少含一个 `.md` 文件（规约同步硬约束的机器侧体现） | ERROR |
| 5 | `tasks.md` 中全部 `[ ]` 已勾选为 `[x]` 时，强校验 `review.md` 存在且结论为"通过" | ERROR |
| - | 任一 ERROR | 退出码 1 |
| - | 仅 WARN（如目录名含大写、tasks 未全勾但 review 已存在） | 退出码 0 |

### 输出与退出码

- 彩色 `OK` / `WARN` / `ERROR` 三档输出，每个变更目录一段
- 末尾汇总 `Summary: X OK / Y WARN / Z ERROR`
- 退出码：任一 ERROR → `1`，否则 `0`

### 边界与不做的事

- **不做语义判断**：不读 proposal/design 内容是否合理，不评估 tasks 拆分粒度
- **不替代代码审查**：脚本仅校验制品**存在与命名**，`review.md` 是否真正逐项核对依然由人/AI 主观保证
- **不强制运行测试**：测试由 `npm test` / `npm run test:golden` 各自负责

### 与阶段闸门的关系

- **Phase 1 退出前** 必须 `npm run check:workflow` 全绿（保证 proposal/design/tasks/specs 齐备）
- **Phase 4 归档前** 必须再次 `npm run check:workflow` 全绿（保证 tasks 全勾 + review 通过）
- 闸门表里"机器可校验"的格子统一由本脚本兜底，AI/人不再口头声明"我检查过了"

### 演进策略

- 新增检查项时，必须保持**零依赖**与**只检查硬约束**两条底线
- 涉及规则放宽/收紧的，须走 OpenSpec 变更流程修改本节，不得直接改脚本

## 流程纪律（新增于 2026-06-03）

### 教训：Bug 报告必须先提案，后修复

**2026-06-03 事件回顾：** 用户报告管理页面空白渲染 bug 后，Claude 跳过提案创建直接进入调试和修复，导致修复代码被直接提交而非通过正式变更流程处理。

**暴露的问题：**
- 收到 bug 后直觉反应是"找到根因 → 修复代码"，跳过了"创建变更提案 → 规划修复"的强制步骤
- 错误的心理模型：小改动 = 不需要走流程
- 修复代码和测试已经写好，但 OpenSpec 变更尚未创建，流程倒挂

### 强制规则

**接收到任何 Bug 报告后，执行顺序固定为：**

```
第 1 步：创建 Bug 修复变更提案（/opsx:ff v<MAJOR>-<MINOR>-<PATCH>-<name>）
  → 这是第一步，不得先写代码再建提案
第 2 步：完成 proposal / design / specs / tasks
第 3 步：按 tasks 执行（TDD：RED → GREEN）
第 4 步：代码审查
第 5 步：归档
```

**任何情况下不得颠倒步骤 1 和步骤 3 的顺序。** 即使修复代码已经写完、测试已经通过，也必须先创建变更、完成制品，再提交代码。流程不可倒挂。

### 识别信号

以下信号意味着你正在跳步：

| 信号 | 正确的做法 |
|------|-----------|
| "这个 bug 很简单，我先看看代码" | 先创建变更提案 |
| "根因我已经找到了，直接改吧" | 先创建变更提案 |
| "修复代码写好了，补个提案吧" | 先创建变更提案（即使代码已写好） |
| "只改一行，不用走流程" | 必须走流程，一行也要提案 |

**发现代码已写好但提案未创建时：** 不得直接提交。必须先创建变更提案并完成所有制品，将现有代码作为实施内容，再提交。流程正确优先于效率。

## 版本号规则

遵循 **[语义化版本 2.0.0（SemVer）](https://semver.org/)** 标准。

### 强制要求

**所有变更提案必须包含版本号。** 版本号是变更目录名的固定组成部分，不可省略。

### 版本格式

```
<MAJOR>.<MINOR>.<PATCH>
```

三段式非负整数，初始提案始终为 `1.0.0`。

### 递增规则

| 变更类型 | 说明 | 示例 |
|----------|------|------|
| **MAJOR** | 新增功能模块、重大重构、breaking change | `1.0.0` → `2.0.0` |
| **MINOR** | 功能增强、非破坏性新增（不涉及规范变更） | `1.0.0` → `1.1.0` |
| **PATCH** | Bug 修复、样式调整、文案修改 | `1.0.0` → `1.0.1` |

递增高级位时低位归零：`1.2.3` → MAJOR 递增 → `2.0.0` → MINOR 递增 → `2.1.0`。

### 变更目录命名规则

```
openspec/changes/v<MAJOR>-<MINOR>-<PATCH>-<kebab-name>/
```

- 三段版本号用连字符 `-` 分隔（兼容 openspec 工具限制）
- `<kebab-name>` 为英文短横线命名法，简要描述变更
- **proposal 标题使用中文描述变更内容**

| 示例 | 说明 |
|------|------|
| `v1-0-0-salary-calculator` | 首个功能提案，标题 `[V1.0.0] 实时工资计算器` |
| `v1-1-0-export-data` | MINOR 增强：导出数据，标题 `[V1.1.0] 数据导出功能` |
| `v1-0-1-fix-salary-bug` | PATCH 修复：工资计算 Bug，标题 `[V1.0.1] 修复工资计算错误` |

### 提案标题标注

在 `proposal.md` 标题中用 **中文** 标明完整版本号和变更名称：

```markdown
# [V1.0.0] 实时工资计算器
```

版本号部分使用点号 `.`（标准 SemVer 格式），仅在目录名中替换为连字符。

### 注意事项

- 变更目录创建时必须确定版本号，后续不可修改
- 主规范（`openspec/specs/`）不携带版本号，版本号仅属于变更生命周期
- 同一次变更内的所有制品（proposal / design / specs / tasks）使用同一个版本号
- 归档后版本号随变更记录存档，不继续递增
