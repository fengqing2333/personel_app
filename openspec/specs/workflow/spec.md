# OpenSpec 与 Superpowers 协作规范

## 总则

**不得因任何侥幸心理跳过强制步骤。** 以下场景均不构成跳过理由：
- "这个 Bug 很简单，不需要写测试"
- "只改了一行注释，不需要审查"
- "这个是紧急修复，先合并再说"
- "上次也是这样改的，没问题"
- "测试用例太多了，跑不过来"

**任何跳过强制步骤的行为，必须由用户在对话中明确授权，且事后补全所有遗漏步骤。** Claude 不得自行决定跳过。

---

## 三条铁律

```
铁律一：Bug 修复必须失败用例先行
  → 禁止不写测试直接修复代码

铁律二：归档前必须代码审查
  → 禁止跳过审查直接归档

铁律三：审查发现问题必须创建新提案修复
  → 禁止在原变更中修补主要/阻断问题
```

**以上三条铁律在本项目强制执行，无例外。**

---

## 核心理念

```
OpenSpec     → 规定"做什么"（需求规格层）
Superpowers  → 确保"做好"（工程流程层）
```

- **OpenSpec** 产出持久化变更制品：`proposal.md` → `design.md` → `specs/` → `tasks.md`
- **Superpowers** 产出工程流程制品：设计方案、实施计划、验证证据、项目记忆

## 协作流程

### 完整流程

```
┌──────────────────────────────────────────────────────────┐
│ Phase 1: 探索 (Superpowers)                               │
│   ├── 读取项目记忆和环境                                    │
│   ├── 逐条澄清需求                                          │
│   └── 输出：2-3 方案对比 + 推荐方向 + 设计草案               │
│   交付至：docs/superpowers/specs/                           │
└──────────────────────┬───────────────────────────────────┘
                       ↓ 方向确认
┌──────────────────────────────────────────────────────────┐
│ Phase 2: 规格 (OpenSpec)                                  │
│   ├── /opsx:new 创建变更脚手架                             │
│   ├── proposal.md  → 为什么做、做什么                       │
│   ├── design.md    → 技术方案、权衡                        │
│   ├── specs/       → 规范性需求、场景                      │
│   └── tasks.md     → 可执行实施清单                        │
│   交付至：openspec/changes/<change-name>/                   │
└──────────────────────┬───────────────────────────────────┘
                       ↓ 用户审批
┌──────────────────────────────────────────────────────────┐
│ Phase 3: 执行 (Superpowers)                               │
│   ├── 编写实施计划到 docs/superpowers/plans/               │
│   ├── 创建隔离工作区 (worktree)                             │
│   ├── TDD: RED → GREEN → REFACTOR                         │
│   ├── Bug 修复：必须先写失败用例，再写修复代码               │
│   └── 回归验证（必须输出新鲜命令结果）                       │
└──────────────────────┬───────────────────────────────────┘
                       ↓ 验证通过
┌──────────────────────────────────────────────────────────┐
│ Phase 3.5: 代码审查 (Code Review)                         │
│   ├── 审查是归档的前置条件，不可跳过                         │
│   ├── 发现主要/阻断问题 → 创建新提案，返回 Phase 2          │
│   ├── 发现次要问题 → 就地修复，仍需通过审查                  │
│   └── 审查通过 → 进入归档                                 │
└──────────────────────┬───────────────────────────────────┘
                       ↓ 审查通过
┌──────────────────────────────────────────────────────────┐
│ Phase 4: 归档 (OpenSpec)                                  │
│   ├── /opsx:archive 归档变更、合并主规范                    │
│   └── run-superpowers-memory-closeout (如启用)             │
└──────────────────────────────────────────────────────────┘
```

### 路径规则

**小变更（需求明确）：**
```
/opsx:propose → /opsx:apply → 代码审查 → /opsx:archive
```

**复杂变更（需求模糊 / 跨模块影响）：**
```
Superpowers 探索 → OpenSpec 规格 → Superpowers 执行
  → 代码审查 → OpenSpec 归档
```

两条路径都必须经过代码审查。无例外。

---

## 铁律一：Bug 修复必须失败用例先行

### 强制步骤

```
第 1 步：定位 Bug，确认根因
第 2 步：编写失败测试用例（RED）        → 必须写，不可跳过
第 3 步：执行测试，确认测试确实失败      → 验证测试本身有效
第 4 步：编写修复代码使用例通过（GREEN）
第 5 步：执行全部测试，确认无回归
第 6 步：重构优化（REFACTOR）
第 7 步：进入代码审查
```

### 禁止行为

- 禁止不写测试直接修复
- 禁止先修复代码再补测试
- 禁止只做人工验证，不写自动化测试
- 禁止跳过 RED 步骤直接写修复代码

### 异常处理

如果现有测试框架不支持自动化测试（如尚无测试框架），必须先建立测试基础设施，再按上述步骤修复。

---

## 铁律二：归档前必须代码审查

### 触发条件

**所有变更**在归档前必须经过代码审查。即使只改了一行代码。

### 审查流程

```
第 1 步：执行 /superpowers:requesting-code-review 发起审查
第 2 步：按以下标准分类问题：
   - 阻断（Blocker）— 功能错误、安全漏洞、数据丢失
   - 主要（Major）— 设计缺陷、边界遗漏、测试覆盖不足
   - 次要（Minor）— 命名、风格、注释
第 3 步：根据分类执行对应处理
```

### 处理规则

| 审查结论 | 必须执行 |
|----------|----------|
| 无问题 | 进入 Phase 4 归档 |
| 次要问题 | 就地修复，修复后仍需通过审查，然后归档 |
| 主要/阻断问题 | 不得在原变更中修补，必须创建新提案修复（见铁律三） |

---

## 铁律三：审查发现问题必须创建新提案修复

### 适用条件

当审查发现 **主要（Major）** 或 **阻断（Blocker）** 问题时。

### 强制流程

```
第 1 步：/opsx:new 创建新变更（命名格式：fix-review-<原变更名>）
第 2 步：完成 proposal.md → 明确要修复的问题
第 3 步：完成 design.md → 描述修复方案
第 4 步：完成 specs/ → 更新相关规范
第 5 步：完成 tasks.md → 可执行修复任务
第 6 步：用户审批
第 7 步：修复实施（必须遵循铁律一）
第 8 步：修复完成后再次进入代码审查（铁律二）
第 9 步：审查通过后归档修复变更
```

### 禁止行为

- 禁止在原变更中附加修补提交
- 禁止以"顺便修一下"的方式处理审查发现的问题
- 禁止跳过提案直接写修复代码

## 路径选择规则

| 场景 | 强制路径 | 原因 |
|------|----------|------|
| 新功能，需求明确 | OpenSpec 核心路径 | 需求已知，无需探索 |
| 新功能，需求模糊 | 完整四阶段 | 必须先探索确认需求 |
| Bug 修复，根因已知 | OpenSpec 核心路径（必须失败用例先行） | 范围可控，但测试不可少 |
| Bug 修复，根因未知 | Superpowers 探索 → OpenSpec 规格 | 必须先定位才能定方案 |
| 重构，范围明确 | OpenSpec 核心路径 | 行为不变，只需规格化 |
| 重构，影响面广 | 完整四阶段 | 必须多方案评估 |

## 职责边界

| 维度 | OpenSpec | Superpowers |
|------|----------|-------------|
| 需求定义 | ✅ proposal.md — 目标和边界 | ❌ 不参与 |
| 技术设计 | ✅ design.md — 架构决策 | ✅ 设计草案 — 多方案对比 |
| 规范标准 | ✅ specs/ — 活文档 | ❌ 不参与 |
| 任务分解 | ✅ tasks.md — 执行清单 | ✅ 实施计划 — 拆分为子任务 |
| 代码实现 | ❌ 不执行代码 | ✅ TDD 实施、失败用例先行 |
| 代码审查 | ❌ 不参与 | ✅ 审查请求、问题分级 |
| 质量验证 | ❌ 不验证 | ✅ 回归验证、输出新鲜证据 |
| 知识持久化 | ✅ 变更归档到主规范 | ✅ 项目记忆持续更新 |

---

## 补充规则

1. **OpenSpec 不执行代码**：`tasks.md` 完成后，实施由 Superpowers 执行
2. **Superpowers 不写需求文档**：需求必须锁定在 OpenSpec 制品中
3. **单向依赖**：Superpowers 执行以 `tasks.md` 为唯一输入，不得反向修改
4. **规格即真相**：所有偏离需求文档的实现，必须先更新 OpenSpec 制品再实施
5. **新鲜证据**：禁止无命令输出的"我觉得没问题"式完成声明。验证必须输出新鲜命令结果
6. **不可跳步**：禁止跳过审核直接写代码，禁止跳过审查直接归档，禁止跳过提案直接修补
