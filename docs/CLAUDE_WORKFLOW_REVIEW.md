## 0. 审视范围说明

本次审视只关心一个问题:

> **当前规范是否足以保证"每次执行都走完整流程,且代码与规约始终同步"?**

不在范围内的(本报告**不**讨论):
- 紧急修复 / 事后追认通道
- 变更回滚 / 放弃流程
- AI 输出格式契约
- 流程之外的优化项

## 一、整体评价

### 1.1 现有规范的优点

- **三条铁律**用极短口号锁定红线,不易遗忘 ✅
- **OpenSpec ↔ Superpowers 职责边界**清晰,角色不混淆 ✅

### 1.2 关键缺口(围绕"不跳步、规约同步")

| 缺口 | 后果 |
|---|---|
| 规约信息源有两份(CLAUDE.md / workflow spec) | 规约本身就难同步 |
| 流程节点没有"准入/准出"硬校验 | 跳步只能靠自律,无机制兜底 |
| "审查通过"无客观判定标准 | 容易被默认通过 |
| 黄金用例 L1 无法实际执行 | 回归验证形同虚设 |
| 没有强制制品检查清单 | 制品缺失也能进入下一阶段 |

## 二、为"不跳步 + 规约同步"服务的优化建议

### 1. 消除 CLAUDE.md 与 workflow spec 的内容重叠 🔴 P0

**问题**: 三条铁律、流程图、路径选择在两份文件中各写一遍。任意一份被改,另一份就会落后,**规约自身就先违反了"同步"原则**。

**建议**:
- [`CLAUDE.md`](CLAUDE.md) 收敛为 3 块内容:
  1. 项目一句话简介
  2. **唯一指向**: "完整流程规范以 [`openspec/specs/workflow/spec.md`](openspec/specs/workflow/spec.md) 为准,本文件不重复其中任何条款"
  3. 项目结构与技术栈 (项目级元信息)
- 三条铁律、流程图、版本号规则**只在 workflow spec 中维护**。

→ 收益: 规约只有一处真相,从根上避免规约不同步。

---

### 2. 为每个流程节点定义"准入/准出"硬制品 🔴 P0

**问题**: 当前流程图只描述"应该做什么",但**没有规定每个 Phase 必须有什么文件存在才允许进入下一步**。这正是"跳步"能发生的根本原因 — 阶段之间没有闸门。

**建议**: 在 workflow spec 中新增一张 **"阶段闸门表"**,作为强制约束:

| Phase | 进入前必须存在 | 退出前必须存在 |
|---|---|---|
| Phase 1 探索 | (无) | `docs/superpowers/specs/<change>.md` |
| Phase 2 规格 | 上一阶段制品 | `proposal.md` + `design.md` + `specs/` + `tasks.md` 全部存在且非空 |
| Phase 3 执行 | tasks.md 存在 | 所有 tasks 勾选完成 + 测试全绿命令输出 |
| Phase 3.5 审查 | 测试全绿证据 | `review.md` 存在且结论=通过 |
| Phase 4 归档 | review.md 通过 | 主规范已合并 |

→ 收益: 把"不得跳步"从口号变成**可逐项核对的清单**。

---

### 3. 引入 `review.md` 制品,让"审查通过"有客观依据 🔴 P0

**问题**: [`workflow/spec.md`](openspec/specs/workflow/spec.md:139) 的铁律二只规定"必须审查",但没有规定**审查的输出物**。结果是:

- "无问题"和"次要问题"的边界靠主观判断
- 审查结论散落在对话里,事后无从追溯
- 归档命令无法机器校验"是否审查过"

**建议**: 在每个变更目录内强制要求 `review.md`,采用固定模板:

```markdown
# Code Review — v<version>-<name>
- 审查时间:
- 审查范围: (commit / 文件清单)
- 检查清单:
  - [ ] 测试覆盖关键路径与边界
  - [ ] 命名 / 风格符合既有约定
  - [ ] 无遗留 TODO / console.log
  - [ ] specs 与实现一致 (规约同步)
  - [ ] tasks.md 全部勾选
- 发现问题:
  - Blocker:
  - Major:
  - Minor:
- 结论: ☐ 通过  ☐ 就地修复 (Minor)  ☐ 创建新提案 (Major/Blocker)
```

`/opsx:archive` 检测到本目录无 `review.md` 或结论非"通过"时**直接拒绝归档**。

→ 收益: 把"审查通过"从主观感受变成可校验文件,自然地堵住"跳过审查"的口子。其中"specs 与实现一致"一项,直接对应**规约同步**诉求。

---

### 4. 落地 L1 黄金用例的可执行命令 🟠 P1

**问题**: [`workflow/spec.md`](openspec/specs/workflow/spec.md:115) 规定 L1 用 `// @golden:` 注释标记,但 vitest 默认**不识别注释筛选**,[`package.json`](package.json) 也没有 `test:golden` 脚本。**所谓 L1 命令在工程中根本无法执行**,导致铁律一的回归验证步骤实际上每次都被绕过。

**建议** (二选一):
- 用文件名标记: `*.golden.test.js`,在 vitest 中通过 `include` 匹配,新增脚本 `"test:golden": "vitest run --include '**/*.golden.test.js'"`
- 或用 describe 前缀 `[GOLDEN]` + `--testNamePattern '\\[GOLDEN\\]'`

→ 收益: 让"先 L1 后归档"成为真实可执行步骤,而不是规范文档里的死字。

---

### 5. 制品-代码同步硬约束:specs 改动必须出现在同一变更内 🟠 P1

**问题**: 当前规范虽然要求 "规格即真相",但没有定义**如何判断规约和代码同步**。开发者可能修改了 `src/` 而忘记同步 `openspec/changes/<v>/specs/`,审查时也未必能发现。

**建议**: 在 `review.md` 检查清单中显式包含:
- [ ] 本次变更涉及的功能,在 `specs/` 中已有对应描述
- [ ] `src/` 的实现行为与 `specs/` 一致(关键场景逐条对照)
- [ ] tasks.md 中所有项已勾选,且勾选项均能在 commit 中找到对应改动

并在 spec 中追加一条强制条文:

> **任何 `src/` 下的行为变更,必须在同一变更目录的 `specs/` 中有对应条目;如无则视为规约脱节,审查必须打回。**

→ 收益: 从机制上强制规约与代码同进同出。

---

### 6. 自动化校验脚本(可选但强烈建议) 🟠 P1

**问题**: 上述所有"必须 / 不得"目前都依赖人或 AI 自律。一旦疏忽就被绕过。

**建议**: 提供一个轻量脚本 `scripts/check-workflow.mjs`,在 pre-commit / 归档命令中调用,自动校验:

- 当前活跃变更目录是否包含 `proposal.md` / `design.md` / `tasks.md` / `specs/` / `review.md`
- `proposal.md` 标题是否符合 `[Vx.y.z]` 格式
- `tasks.md` 是否所有项已勾选
- `review.md` 结论字段是否为"通过"

**校验失败则禁止提交 / 禁止归档**。

→ 收益: 把整套规约从"AI 自律"升级为"工程闸门",这是确保"不跳步"最可靠的手段。

## 三、优先级与执行顺序

| 优先级 | 任务 | 直接服务的目标 |
|---|---|---|
| 🔴 P0 | (#1) CLAUDE.md 与 workflow spec 去重,单一信息源 | 规约同步 |
| 🔴 P0 | (#2) 阶段闸门表 — 每个 Phase 的准入/准出制品 | 不跳步 |
| 🔴 P0 | (#3) 强制 `review.md` 制品 + 审查清单 | 不跳步 + 规约同步 |
| 🟠 P1 | (#4) L1 黄金用例可执行命令 | 不跳步(回归验证落地) |
| 🟠 P1 | (#5) 制品-代码同步硬约束写入审查清单 | 规约同步 |
| 🟠 P1 | (#6) `check-workflow.mjs` 自动校验 | 不跳步 + 规约同步 (机器兜底) |

**最小落地组合**:`#1 + #2 + #3` 已能解决 80% 的跳步与规约脱节风险;`#6` 是把这套规范从"靠自觉"升级到"靠机器"的关键一跃。

## 四、一句话总结

当前规范的"是否要走流程"已经表达得很清楚,**缺的是"如何机械化地证明流程被走完了"**。引入"阶段闸门表 + review.md 制品 + 校验脚本"这三件事,就能把"不跳过规约"从纪律变成机制。
